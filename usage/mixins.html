Mixins

JSoop provides several fundamental mixins out of the box to help make your application creation easier.

Configurable

Basic Usage

The configurable mixin is used, as it name suggests, to configure classes. At its core, it takes a given config object and applies it to its class.




    
Configurable overrides

    
When applying a config to an object, Configruable overrides anything that already exists in the class. This can have undesirable consequences. Often the benefits of having a mutable class outweigh the pitfalls, but caution should still be taken.




Default Values


While the base of Configurable is fairly powerful on its own, it also gives you the ability to add default config values to your class. This is done by setting using the config property's defaults object. Any value that appears in the default will be applied to the config if it does not exist.


Another powerful tool to remember is that defaults are inherited. Should a subclass' defauls override its parents, then the child's values will take precedence. This means that you can create a chain of default values through your class hierarchy giving you a lot of control over how your classes are configured.





    
Only missing items will be defaulted

    
Configurable will only apply a default value if it doesn't exist in the config given to it. This means that values like undefined and null will not trigger a default value if passed in a config.




Required Config


While default values solve most cases of needed configuration, sometimes you need to be able to force the user to provide specific config values to make your class work. These values can be defined in the required property. If, after apply defaults, a config does not contain all required config items then an error will be thrown. As with defaults, required config items are inherited.



                
Plugin Manager


Plugin Manager is a mixins that allows your classes to create and destroy plugins. The initial setup of plugins will iterate through the plugins property of your class, creating a plugin for each key found.


Each plugin receives a reference to itss owner as part of its config object. This allows the plugin to interact with its owner any way it wants to. A very easy and flexible way to do this is by using the Observable mixin.


It is important to call destroyAllPlugins during your class' teardown phase. This will destroy any plugins it is currently managing. Doing so will avoid memory leaks.



                
Observable


Basic Usage


The observable mixin gives your class the capability to manage custom events. You can fire a custom event using the fireEvent method, passing it a name as well as any arguments that are associated with that event.


During setup, observable will automatically bind any event listeners defined in the listeners property of the object. This gives it great synergy when used with Configurable.





    
Teardown is required

    
It is extremely important that removeAllListeners and removeAllManagedListeners are called during an object's teardown phase. Failing to do so can cause significant memory leaks.




Multiple Listeners


You can add or remove multiple listeners at once using the object format. This is done by passing an object with event names as the keys and listeners as the values. Default versions of options can be added to any level of the object to apply to either all or a single listener.




Managed Listeners


For a cleaner way to listen to another object's events, you can use managed listeners. These have the same syntax as traditional listeners but use the mon and moff methods instead. The only difference being that you need to pass the object you want to listen to as your first parameter.
